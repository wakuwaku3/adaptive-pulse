# 要件

## 目的・背景

減量中 (92kg → 75kg 目標) のユーザ (39 歳・男性・エンジニア) が毎朝ジムの筋トレ後にクロスストレーナーで行う有酸素インターバルを、固定時間 (60 秒速く / 30 秒遅く) から**心拍数トリガーのアダプティブ方式**に置き換える。既製の Wear OS アプリには心拍閾値でフェーズが切り替わるインターバルを組める決定打が無いため、Pixel Watch 向けに自作する。

利用者は本人のみ。Pixel Watch は未購入で、まずエミュレータでロジックを検証し、手応えがあれば購入する。

## セッションのライフサイクル

操作は**開始と (必要時の) 手動停止のみ**。運動中 (腕を振っている) の画面操作は不可能な前提で、フェーズ通知はすべて振動で行う。

1. 開始操作で計測開始。最初のフェーズは高強度。
2. **高強度フェーズ**: 心拍が上限閾値 (155bpm) を超えたら振動 → 回復フェーズへ。
3. **回復フェーズ**: 心拍が下限閾値 (140bpm) を下回ったら振動 → 高強度フェーズへ。
4. 高強度→回復で 1 サイクル。目標サイクル数 (6〜8、デフォルト 7) 到達後、その回復フェーズの完了 (下限閾値を下回る) で終了振動を鳴らし、**計測を自動停止**する。

### フェーズタイムアウト

センサー不調や負荷不足で閾値に到達しないままフェーズが長引いた場合 (高強度デフォルト 3 分 / 回復デフォルト 3 分) は、**疲労ブレーキ扱いでセッション自体を終了する**。閾値に届かない状態でフェーズだけ強制遷移してもセッションの意義を満たさないため、無理に継続しない。

### 履歴の保存ポリシー

| 終わり方 | 保存 |
|---|---|
| 完走 (目標サイクル到達 / 疲労ブレーキ / タイムアウト終了) | 常に保存 |
| 手動停止 | 完走サイクルが 1 以上なら保存 (それ以前は捨てる) |
| 異常終了 (センサー経路の失敗等) | 常に保存 (途中までの結果を残す) |

`cycles` は「高強度→回復まで完走したサイクル数」とする (未完了の進行中サイクルは含めない)。

## 疲労ブレーキ

サイクルを重ねて疲労が蓄積すると、同じ負荷でも心拍が上がりやすくなる (上限到達が速くなる)。これを検知して無理を止める。

- 各サイクルの**高強度所要時間**は「心拍が下限閾値を上向きに超えた時点から、上限閾値に到達するまで」の時間で測る。セッション開始直後のウォームアップ区間 (低心拍から下限閾値までの上昇) は計測に含まれないため、初回サイクルと 2 回目以降を公平に比較できる。固定時間による区別は使わない。
- **基準時間** = 初回サイクルの高強度所要時間。ただし最低基準時間 (デフォルト 45 秒) に満たない場合 (筋トレ直後で心拍が既に下限閾値付近の状態で開始した場合など) は妥当な基準とみなさず、2 サイクル目を基準にする。
- 現在サイクルの高強度所要時間 ≤ 基準時間 × 早期終了係数 (0.5) になったら疲労とみなす。
- 発動時: 専用の振動パターンで通知し、**現在のサイクルを最終サイクルとする** (その回復フェーズ完了でセッション終了)。画面操作なしで完結する。

## 設定値

すべてハードコードせず変更可能にする。DataStore に永続化し、ウォッチ内の設定画面 (一覧 + Stepper 編集) から変更する。閾値の「上限 > 下限」は編集 UI のレンジで構造的に保証する。

| 設定 | デフォルト | 備考 |
|---|---|---|
| 上限閾値 | 155bpm | 高強度→回復の遷移トリガー |
| 下限閾値 | 140bpm | 回復→高強度の遷移トリガー |
| 目標サイクル数 | 7 | 6〜8 を想定 |
| 早期終了係数 | 0.5 | 基準時間に対する比率 |
| 最低基準時間 | 45 秒 | 初回サイクル基準の妥当性ガード |
| 高強度フェーズ上限時間 | 3 分 | タイムアウトで疲労ブレーキ終了 |
| 回復フェーズ上限時間 | 3 分 | タイムアウトで疲労ブレーキ終了 |

### 閾値初期値の自動導出 (拡張)

155/140 はユーザ本人 (39 歳) の体感ベースの値。年齢から最大心拍数を推定する Tanaka 式 (HRmax = 208 − 0.7 × 年齢) に当てはめると 155 ≒ 86% HRmax、140 ≒ 77% HRmax に相当するため、初期値は「年齢 → %HRmax」で導出する (上限 86% / 下限 77%)。

- 年齢 (生年) は初回に手入力する。Health Connect に年齢・性別のプロファイル型は無い想定 (実装時に要確認)。
- 心拍ゾーンを決める主因は年齢と安静時心拍であり、身長・体重はゾーン算出に使わない。安静時心拍は Health Connect から読めるため、将来 Karvonen 式 (心拍予備能ベース) への切り替え余地を残す。

## UI 方針

- 表示言語は英語 (運動中の一瞥で読める短い単語が中心のため。スタイル重視のユーザ嗜好)。
- 黒背景 + フェーズの直感色分け (高強度=コーラル / 回復=ミント / 完了=ゴールド / ウォームアップ=クールブルー)。心拍数字は最大サイズで表示する。
- セッション開始直後、心拍が下限閾値を上向きに超えるまでは「WARM-UP」と表示する (計測対象外の区間であることを画面が正しく伝える)。
- ウォッチフェイスのタイルから 1 タップでセッションを開始できる (毎朝の起動摩擦を最小化)。
- 詳細なトーンは `.claude/rules/ui.md`。

## 計測メトリクス

- **消費カロリー**: Health Services の CALORIES_TOTAL をセッション中・終了画面に表示する (減量目的の直接的なフィードバック)。取得には ACTIVITY_RECOGNITION 許可が必要で、拒否時はカロリー無しで計測を継続する。
- **ゾーン滞在率**: 下限〜上限の帯にいた時間の割合を終了画面に表示する (トレーニング品質の指標)。

## 振動パターン

イベントごとに体感で区別できる別パターンにする: 高強度→回復 / 回復→高強度 / 疲労ブレーキ / セッション終了。具体パターンは実機で調整する (タイムアウトでの強制終了は単発の「セッション終了」振動として通知され、終了画面で `fatigueBrake=true` がわかる)。

## 非機能要件

- 個人利用のみ。一般公開はしない。配布は Google Play の**内部テストトラック** (本人のみテスター登録) を使い、ウォッチへは Play 経由でインストール・自動更新する。Play 開発者登録は支払い済み (2026-06)。内部テストには新規個人アカウントの「テスター 12 人 × 14 日」要件は適用されない (あれは製品版公開のゲート)。開発中の即時反映には Wi-Fi ADB サイドロードを併用する。
- 1 セッションは 12〜15 分。画面オフ中も計測を継続する (Foreground Service + Ongoing Activity)。
- 手首の光学式心拍センサーは腕を振る運動で 10〜30 秒の遅延・ノイズが出る前提で設計する:
  - 閾値 155/140 の 15bpm 幅は意図的なヒステリシス (チャタリング防止)。
  - 必要に応じて移動平均などの平滑化を入れる。
  - 根本対策は Polar H10 接続 (拡張)。

## 拡張 (MVP 後)

- セッション結果 (時間・サイクル数・心拍推移) の Health Connect 書き込み。
- Polar H10 (胸ストラップ) を BLE 標準 Heart Rate Profile (GATT, Service 0x180D) で接続し、手首光学センサーの代替にする。
- 閾値初期値の年齢ベース自動導出 (上記)。
- セッション履歴と体力トレンド (高強度所要時間・HRR の推移)。可視化はスマートフォン向け画面を本リポジトリの別モジュール (同一 applicationId の phone APK。Play では同一アプリの別フォームファクター) として作り、Wearable Data Layer でウォッチから転送する。

## 開発の進め方

1. **エミュレータで最小版ロジックを作る** (実機不要。Wear OS エミュレータは心拍の合成データを流せる)。ステートマシン (高強度/回復/終了)・サイクルカウント・疲労ブレーキまで。ロジックは UI と分離して単体テストを書く。
2. 手応えがあれば Pixel Watch を購入し、最小版 (心拍表示 + 閾値振動) をジムで実戦投入。
3. 使いながら拡張 (Health Connect 書き込み・設定画面) を追加。
4. 手首センサーの遅延が実用上問題なら Polar H10 を購入して BLE 対応を追加。

## コスト前提

開発・API・サイドロードはすべて無料。同期 (履歴・設定) は Firebase Spark 枠 (Auth + Firestore + Rules) だけで賄い、billing アカウントはリンクしない。Google Play 公開時のみ登録料 25 ドルだが、公開しないため不要。
