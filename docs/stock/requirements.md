# 要件

## 目的・背景

減量中 (92kg → 75kg 目標) のユーザ (39 歳・男性・エンジニア) が毎朝ジムの筋トレ後にクロスストレーナーで行う有酸素インターバルを、固定時間 (60 秒速く / 30 秒遅く) から**心拍数トリガーのアダプティブ方式**に置き換える。既製の Wear OS アプリには心拍閾値でフェーズが切り替わるインターバルを組める決定打が無いため、Pixel Watch 向けに自作する。

利用者は本人のみ。Pixel Watch は未購入で、まずエミュレータでロジックを検証し、手応えがあれば購入する。

## セッションのライフサイクル

操作は**開始と (必要時の) 手動停止のみ**。運動中 (腕を振っている) の画面操作は不可能な前提で、フェーズ通知はすべて振動で行う。

1. 開始操作で計測開始。最初のフェーズは高強度。サイクルカウントは 0 から始まる。
2. **高強度フェーズ**: 心拍が上限閾値 (Karvonen 86%, age 39 / rest 60 で 164bpm) を超えたら振動 → 回復フェーズへ。**この遷移のタイミングでサイクルカウントを +1** する。
3. **回復フェーズ**: 心拍が下限閾値 (Karvonen 60%, age 39 / rest 60 で 133bpm) を下回ったら振動 → 高強度フェーズへ。
4. 目標サイクル数 (6〜8、デフォルト 7) ぶん「高強度→回復」を回し、最終サイクルの回復フェーズ完了 (下限閾値を下回る) で終了振動を鳴らし、**計測を自動停止**する。

### フェーズタイムアウト

センサー不調や負荷不足で閾値に到達しないままフェーズが長引いた場合 (高強度デフォルト 4 分 / 回復デフォルト 4 分) は、**疲労ブレーキ扱いでセッション自体を終了する**。閾値に届かない状態でフェーズだけ強制遷移してもセッションの意義を満たさないため、無理に継続しない。タイムアウト 4 分は HRR (1 分目 ~25-30bpm 降下後プラトー: Imai 1994, Cole 1999) と Karvonen 0.86/0.60 のギャップ ~30bpm から逆算した値で、後半サイクルでも届きうる余裕を持たせる。

### 履歴の保存ポリシー

| 終わり方 | 保存 |
|---|---|
| 完走 (目標サイクル到達 / 疲労ブレーキ / タイムアウト終了) | 常に保存 |
| 手動停止 | 完走サイクルが 1 以上なら保存 (それ以前は捨てる) |
| 異常終了 (センサー経路の失敗等) | 常に保存 (途中までの結果を残す) |

`cycles` は「高強度→回復まで完走したサイクル数」とする (未完了の進行中サイクルは含めない)。

## 疲労ブレーキ

サイクルを重ねて疲労が蓄積すると、同じ負荷でも心拍が上がりやすくなる (上限到達が速くなる) かつ、回復が遅くなる (副交感神経の再活性が鈍る)。両側のシグナルから検知して無理を止める。

### 高強度短縮 (上限到達までの時間が短くなる)

- 各サイクルの**高強度所要時間**は「心拍が下限閾値を上向きに超えた時点から、上限閾値に到達するまで」の時間で測る。セッション開始直後のウォームアップ区間 (低心拍から下限閾値までの上昇) は計測に含まれないため、初回サイクルと 2 回目以降を公平に比較できる。固定時間による区別は使わない。
- **基準時間** = 初回サイクルの高強度所要時間。ただし最低基準時間 (デフォルト 45 秒) に満たない場合 (筋トレ直後で心拍が既に下限閾値付近の状態で開始した場合など) は妥当な基準とみなさず、2 サイクル目を基準にする。
- 現在サイクルの高強度所要時間 ≤ 基準時間 × 早期終了係数 (デフォルト 0.5) になったら疲労とみなす。
- 発動時: 専用の振動パターン (`FatigueBrake` イベント) で通知し、**現在のサイクルを最終サイクルとする** (その回復フェーズ完了でセッション終了)。画面操作なしで完結する。

### 回復遅延 (下限到達までの時間が長くなる)

学術的には HRR (Heart Rate Recovery) の鈍化として自律神経指標の根拠が厚い (Cole et al. 1999, Imai et al. 1994, Buchheit & Laursen 2013)。

- 各サイクルの**回復所要時間**は「上限閾値に到達した時点 (= 回復フェーズ開始) から、下限閾値を下回るまで」の時間で測る。
- **回復基準時間** = 高強度基準が確定したサイクルと同じサイクルの回復時間 (高強度側の minBaseline ガードに連動させ、歪んだ初回計測を避ける)。
- 現在サイクルの回復所要時間 ≥ 回復基準時間 × 回復疲労係数 (デフォルト 1.5) になったら疲労とみなし、次サイクルには進めずセッション終了。発動はサイクル完了の瞬間のため、`SessionFinished` 振動 + `fatigueBrake=true` フラグで履歴に残す (タイムアウト経路と同じ扱い)。

## 設定値

すべてハードコードせず変更可能にする。DataStore に永続化し、ウォッチ内の設定画面 (一覧 + Stepper 編集) から変更する。閾値の「上限 > 下限」は編集 UI のレンジで構造的に保証する。

| 設定 | デフォルト | 備考 |
|---|---|---|
| 年齢 | 39 歳 | Karvonen 式の入力。Tanaka 式で最大心拍に変換する |
| 安静時心拍 | 60bpm | Karvonen 式の入力。Health Connect 連携 ON のとき日次自動更新 |
| 上限閾値 | Karvonen 86% | デフォルト = 安静時心拍 + 0.86 × 心拍予備能 (≒91% HRmax, T@VO₂max 帯)。明示指定で上書き可 |
| 下限閾値 | Karvonen 60% | デフォルト = 安静時心拍 + 0.60 × 心拍予備能 (≒70% HRmax, 長尺 HIIT の回復目標帯)。明示指定で上書き可 |
| 目標サイクル数 | 7 | 6〜8 を想定 |
| 早期終了係数 | 0.5 | 高強度基準時間に対する比率 |
| 回復疲労係数 | 1.5 | 回復基準時間に対する比率 (HRR 鈍化判定) |
| 最低基準時間 | 45 秒 | 初回サイクル基準の妥当性ガード |
| 高強度フェーズ上限時間 | 4 分 | タイムアウトで疲労ブレーキ終了 |
| 回復フェーズ上限時間 | 4 分 | タイムアウトで疲労ブレーキ終了 |

### セッション中の閾値ナッジ

その日のコンディションに合わせて手元で軽く調整できるよう、セッション中に**現フェーズが見ている閾値**を ±1bpm 単位で動かせる。

- 対象: 高強度フェーズ (WARM-UP 含む) では上限閾値、回復フェーズでは下限閾値。
- 操作: クラウン回転 (1 detent = ±1bpm) と、画面上の左右タッチボタン (`−` / `+`) のどちらでも同じ調整ができる。1 操作ごとに極短の振動 tap で確認される。
- 制約: 上限 > 下限 + 最小ギャップ 5bpm を常に保つよう clamp する。
- 永続化: セッションスコープのみで DataStore に書き戻さない (恒久設定を変えたいときは設定画面から編集する)。
- 表示: Running 画面に `▲ upper  ▼ lower` を常時表示し、現フェーズで動かしている側を色で示す。

### 閾値初期値の自動導出

上限・下限閾値の初期値は年齢と安静時心拍から Karvonen 式 (心拍予備能ベース) で導出する。

- 最大心拍は Tanaka 式 (HRmax = 208 − 0.7 × 年齢, Tanaka et al. 2001) で推定する。
- 目標心拍 = 安静時心拍 + 強度 × (HRmax − 安静時心拍)。デフォルト強度は上限 86% / 下限 60% (Norwegian 4×4 等の長尺 HIIT が回復目標とする 60-70% HRmax 帯に対応; Wisløff 2007, Helgerud 2007, Buchheit & Laursen 2013)。
- 心拍ゾーンを決める主因は年齢と安静時心拍であり、身長・体重はゾーン算出に使わない。
- 年齢は設定画面で編集する。
- 安静時心拍は phone 側の Health Connect 連携が ON のとき日次自動更新する。`HealthSyncWorker` が通常同期 (今日 + 過去 7 日) + 初回 backfill (過去 5 年) のついでに直近の `RestingHeartRateRecord` を取り、`PhoneSync.updateSettingsEverywhere` 経由で LWW 同期する (phone DataStore → 共通 Firestore → watch Data Layer)。phone UI から手動編集することもでき、最後に書いたほうが勝つ。
- 明示指定された上限/下限閾値はそちらが優先される (Karvonen は default に流れるだけで上書きはしない)。HC が安静時心拍だけを更新しても上限/下限は不変なので、ゾーンを追随させたいときは手動で調整するか、後続の「現プロファイルから再計算」アクションを使う。

## UI 方針

- 表示言語は英語 (運動中の一瞥で読める短い単語が中心のため。スタイル重視のユーザ嗜好)。
- 黒背景 + フェーズの直感色分け (高強度=コーラル / 回復=ミント / 完了=ゴールド / ウォームアップ=クールブルー)。心拍数字は最大サイズで表示する。
- セッション開始直後、心拍が下限閾値を上向きに超えるまでは「WARM-UP」と表示する (計測対象外の区間であることを画面が正しく伝える)。
- セッション中は **画面オフを抑止する** (`FLAG_KEEP_SCREEN_ON`)。画面オフ中に Health Services のサンプリングが落ちる挙動を避けるためで、運動中はそもそも画面を見ない時間が多いことより、サンプリング欠落の方を嫌う。
- セッション中は (1) セッション総経過 (2) 現サイクル経過 (3) 現フェーズ経過 の 3 つの時間を同時表示する (運動中にどれくらい走ったか・このサイクルにどれだけ居るか・このフェーズに何分張り付いているかを一瞥で把握するため)。
- ウォッチフェイスのタイルから 1 タップでセッションを開始できる (毎朝の起動摩擦を最小化)。
- 詳細なトーンは `.claude/rules/ui.md`。

## Phone ライブセッション画面

運動中は腕を振っているため watch 画面は読み取れない。phone を機材のコンソール上に置いて視野の端で同期する想定で、セッション中の現在値 (心拍・フェーズ・経過時間・閾値) を phone 画面に常時表示する。

- **配置**: クロストレーナー本体のコンソール上に phone を置き、画面の端に視線を流すだけで読める運用。タップは不要。
- **表示**: 心拍数字を最大サイズで中央に置き、フェーズ色 (高強度=コーラル / 回復=ミント / WARM-UP=クールブルー / 完了=ゴールド) で帯を塗る。下に (1) セッション総経過 (2) 現サイクル経過 (3) 現フェーズ経過、現フェーズが見ている閾値 (▲ upper / ▼ lower)、サイクル進捗 (n/N)、カロリーを並べる。watch 側 Running 画面と同じ情報密度をスマホサイズで提示する。
- **自動起動**: watch でセッションを開始した瞬間、phone を手で開かなくてもライブ画面が前面に出る。Wear → Phone の通知経由で full-screen intent を発火させ、`MainActivity` をライブ画面で立ち上げる。
- **スリープ抑止**: ライブ画面表示中は `FLAG_KEEP_SCREEN_ON` で画面オフを抑止する (watch 側と同じ理由で、画面オフによるサンプル欠落と表示中断を避ける)。
- **状態同期**: watch の Foreground Service が現在値スナップショット (心拍・フェーズ・サイクル・3 経過時間・閾値・カロリー・cadence) を Wearable Data Layer (`DataClient`) に約 1Hz で書き、phone 側 listener が in-memory `StateFlow` に流す。セッション終了時は DataItem を削除して phone のライブ画面を自動で閉じる。
- **操作は watch 側で完結**: 閾値ナッジ・停止操作は watch 側のみ。phone はビューワーに徹し、読みやすさを優先する。
- **権限**: Android 13+ の `POST_NOTIFICATIONS`、Android 14+ の `USE_FULL_SCREEN_INTENT` をユーザに 1 度許可してもらう (初回は通知タップで開く fallback でも動く)。

## 計測メトリクス

- **消費カロリー**: Health Services の CALORIES_TOTAL をセッション中・終了画面に表示する (減量目的の直接的なフィードバック)。取得には ACTIVITY_RECOGNITION 許可が必要で、拒否時はカロリー無しで計測を継続する。
- **ゾーン滞在率**: 下限〜上限の帯にいた時間の割合を終了画面に表示する (トレーニング品質の指標)。

## 振動パターン

イベントごとに体感で区別できる別パターンにする: 高強度→回復 / 回復→高強度 / 疲労ブレーキ / セッション終了。具体パターンは実機で調整する (タイムアウトでの強制終了は単発の「セッション終了」振動として通知され、終了画面で `fatigueBrake=true` がわかる)。

## 非機能要件

- 個人利用のみ。一般公開はしない。配布は Google Play の**内部テストトラック** (本人のみテスター登録) を使い、ウォッチへは Play 経由でインストール・自動更新する。Play 開発者登録は支払い済み (2026-06)。内部テストには新規個人アカウントの「テスター 12 人 × 14 日」要件は適用されない (あれは製品版公開のゲート)。開発中の即時反映には Wi-Fi ADB サイドロードを併用する。
- 1 セッションは 12〜15 分。画面オフ中も計測を継続する (Foreground Service + Ongoing Activity)。
- 手首の光学式心拍センサーは腕を振る運動で 10〜30 秒の遅延・ノイズが出る前提で設計する:
  - 上限と下限の差 (15bpm 程度) は意図的なヒステリシス (チャタリング防止)。
  - 必要に応じて移動平均などの平滑化を入れる。
  - 根本対策は Polar H10 接続 (拡張)。

## 拡張 (MVP 後)

- セッション結果 (時間・サイクル数・心拍推移) の Health Connect 書き込み。
- Polar H10 (胸ストラップ) を BLE 標準 Heart Rate Profile (GATT, Service 0x180D) で接続し、手首光学センサーの代替にする。
- 設定画面に「現プロファイルから Karvonen で上下限を再計算する」アクション。
- セッション履歴と体力トレンド (高強度所要時間・回復所要時間の推移)。可視化はスマートフォン向け画面を本リポジトリの別モジュール (同一 applicationId の phone APK。Play では同一アプリの別フォームファクター) として作り、Wearable Data Layer でウォッチから転送する。

## 開発の進め方

1. **エミュレータで最小版ロジックを作る** (実機不要。Wear OS エミュレータは心拍の合成データを流せる)。ステートマシン (高強度/回復/終了)・サイクルカウント・疲労ブレーキまで。ロジックは UI と分離して単体テストを書く。
2. 手応えがあれば Pixel Watch を購入し、最小版 (心拍表示 + 閾値振動) をジムで実戦投入。
3. 使いながら拡張 (Health Connect 書き込み・設定画面) を追加。
4. 手首センサーの遅延が実用上問題なら Polar H10 を購入して BLE 対応を追加。

## コスト前提

開発・API・サイドロードはすべて無料。同期 (履歴・設定) は Firebase Spark 枠 (Auth + Firestore + Rules) だけで賄い、billing アカウントはリンクしない。Google Play 公開時のみ登録料 25 ドルだが、公開しないため不要。
