# ペース (cadence) メトリクス追加の検討

## 動機

ユーザの実運用上の課題:

> 運動中の適切なペース配分がわからない。メトロノーム的にリズムを刻めばペース配分しやすいのではないか。

現状のメトリクスは HR + カロリーのみで、ペース (クロストレーナーの cadence) は計測も提示もしていない。これを追加すべきか・どう設計するかを検討する。

別セッションで進行中の「watch → phone アプリへの状態同期」と統合する想定で、表現先は **phone アプリ画面** を主に置く (音・光は周囲に迷惑になるため使えない / 運動中の腕は振っていて watch 画面は見られない)。

## そもそもペースを気にする理由

HR トリガー設計の補助として有効。ペース管理そのものが目的ではない。

- **HR の遅延を補う**: 光学式 HR は 10〜30 秒遅延する (要件 §非機能要件)。スタート時に踏み込み過ぎると、HR には反映されない間に過負荷状態に入り、気付いた時には上限に張り付いて回復に時間がかかる。ペースは即時に制御できるので、HR の遅延を先回りで補える。
- **疲労ブレーキの比較を意味あるものにする**: 既存疲労ブレーキは「高強度所要時間」(下限から上限到達までの時間) で判定する (要件 §疲労ブレーキ)。この所要時間は踏み込みの強さに敏感で、今日強く踏めば早く上限到達=疲労扱い、緩く踏めば遅く到達=元気扱い、と誤判定する。ペースを揃えれば負荷が固定され、所要時間差が真に心肺フィットネス・疲労を反映する。
- **副次効果**: 同一ペース下での HR 応答は HR 到達より早く出るため、疲労の早期検知シグナルになり得る (フェーズ B 以降の発展余地)。

逆に、ペース管理を**フェーズ切替のリアルタイムトリガー**には使わない (HR トリガー設計の単純さを壊す)。

## RPS の単位 convention

クロストレーナーで観測される RPS には 2 つの流派がある。

- **「左右 1 サイクル = 1 stride」流派** (機械系の一部)
- **「片足の踏み込み 1 回 = 1 step」流派** (マシン表示・ランニング業界・おそらく Health Services `STEPS_PER_MINUTE` も同じ)

ユーザの実機マシン表示の実測:

| フェーズ | SPM (step) | RPS (step) | 体感 |
|---|---|---|---|
| WARM-UP | ~80 | ~1.3 | 軽くアップ |
| **高強度** | **140〜150** | **2.3〜2.5** | 実測値 |
| **回復** | **~70** | **~1.2** | 実測値 |

→ 本プロジェクトでは **step convention に揃える** (マシン表示および Health Services と一致させる)。

読み取れること:

- 高強度 2.4 Hz ≈ メトロノーム 144 BPM。視覚 pulse として「目で追えるギリ速い」域。
- 回復 1.2 Hz ≈ 72 BPM。安静時心拍テンポで「見ていて落ち着く」域。
- 高強度と回復で **2 倍差** がある。phone 画面の脈動テンポそのもので「今はどちらのフェーズか」を一瞥で伝えられる。
- 回復で 70 SPM = ほぼ歩行域まで落とすのは設計意図 (心拍を 15bpm 落とすために脚を相当緩める必要がある) と整合。

## ペースをどう決めるか

絶対値ではなく **自己ベースライン** が現実的。

- 最初の数セッションは計測のみ (制御しない)。高強度・回復それぞれの cadence の中央値を記録 → これが自然なペース。
- ベースラインが取れたら、それをフェーズ別ターゲットにする。閾値ナッジと同じ思想で、セッション中に ±1 SPM 程度を手元 (クラウン / タッチ) で調整可能にする。

## どう測るか

`STEPS_PER_MINUTE` 優先、ダメなら加速度生信号からの自前検出。

- Pixel Watch の Health Services には `DataType.STEPS_PER_MINUTE` がある。徒歩・ランの「片足着地 = 1 step」と同じ convention のはずで、機材表示とほぼ同じ単位で取れる**可能性が高い**。
- ただしクロストレーナーは足が接地しない動きなので、step として認識されないリスクがある。実機で要検証。
- 自前検出が必要なら生加速度を取り、3〜5 秒窓で短時間 FFT または peak detection。枯れた処理で、JVM 単体テストで合成波形を流して検証できる (ドメインロジックを Android フレームワークから分離する本 repo の方針と合う)。
- **精度より安定性**: 1 秒ごとの瞬時値はノイジーなので、3〜5 秒窓の median をフィルタとして掛ける。

## どう体験として伝えるか (phone 画面)

phone を機材のコンソール上に置いて、視野の端で同期する想定。音・光は使わない。

候補:

- **拍動円 (推奨)**: ターゲット RPS で円が pulse。色で差分を表現 (within=ニュートラル / 速すぎ=ホット / 遅すぎ=クール)。周辺視で読める・実装が軽い。
- **流れるバー**: 左→右にマーカーが等速移動し脚の動きと位相同期させる。位相検出が要るため重い。最初は不要。
- **波形ストリーム**: 過去 10〜30 秒の cadence ライン。終了画面・振り返り向け。運動中は不要。

運動中は **拍動円** を中心にして、ターゲット数字 + 現値 + 簡単なズレ色で十分。位相同期は overkill。

watch 側 UI は変えない (引き続き心拍数字最優先)。

## watch → phone 同期スキーマへの追加フィールド

別セッションで組む watch → phone 状態同期に、ペースのために足すのはこれだけ:

- `currentRps: Double` (3〜5 秒窓の median、Hz)
- `targetRps: Double` (フェーズ別。初期は固定値、Phase B 以降で自己ベースライン化)
- `phase: Warmup | High | Recovery | Done` (既に同期予定のはず)

phone 側は `targetRps` から拍動円の tempo を計算、`currentRps - targetRps` から色判定。

## 段階的ロードマップ

要件の MVP (HR トリガー + 疲労ブレーキ) はそのまま。拡張フェーズで段階的に。

| フェーズ | 内容 | 完了基準 |
|---|---|---|
| **A: 計測のみ** | Health Services の `STEPS_PER_MINUTE` または自前 FFT で cadence 取得・履歴に保存。phone で trend 可視化 | 自然なばらつきが見えてターゲット値・許容幅を決められる |
| **B: ターゲット表示** | phone 画面に target / current を出す (拍動円) | ターゲットに合わせるとセッションが安定する体感が出る |
| **C: adaptive ターゲット** | 自己ベースラインからターゲットを自動算出 | 何セッションかの実測から自動で妥当な target が出る |
| **D: 疲労シグナル化** | 同一 cadence での HR 応答から疲労を早期検知。既存疲労ブレーキの第 3 シグナル | 既存疲労ブレーキより早く・偽陽性少なく検知できる |

Phase A は計測スキーマと加速度処理だけなので低コスト・低リスクで価値情報が取れる。**ここから始めるのが筋**。Phase C 以降は A の観察結果次第で要否を判断 (自然なばらつきが小さければ C 不要)。

## 残課題 / 次セッションへの引き継ぎ

- Health Services の `STEPS_PER_MINUTE` がクロストレーナーで意味のある値を返すか **実機検証** が必要。
  - 返さなければ加速度生信号からの自前検出に切り替え。
- `targetRps` の Phase A 段階の初期値は **固定値で良いか** (例: 高強度 2.4 / 回復 1.2)、それともユーザが設定画面で編集できるべきか。要件 §設定値 の他項目に合わせ、編集可能にするのが自然。
- phone 画面の拍動円デザインは別セッションで具体化 (本ノートでは方針のみ)。
- 要件 §拡張 にペース管理項目を追記するかは **Phase A の実測結果が出てから** (今は flow に留め、stock 昇格は早い)。
