# program / menu 命名の検討経緯

## 2026-07-21 背景

現状は単一設定 + 単一の繰り返し (targetCycles) しか登録できない。設定の異なる繰り返しを組み合わせて 1 つのトレーニング定義にしたい、という要望が出た。まず階層の命名を決める。

## 検討した階層と候補

最小単位は既存の **cycle** (高強度→回復の 1 往復) を維持で即決。requirements・履歴スキーマ (`cycles`, `targetCycles`)・UI ラベルまで一貫しており、変更すると過去データとドキュメント全体に波及するため。

中間層 (同一設定で cycle を N 回繰り返すまとまり) と全体の組で検討:

| 候補 (中間 / 全体) | 評価 |
|---|---|
| block / menu | 最初の提案。「インターバルのブロック」として一般的だが、ユーザの語感に合わず却下 |
| set / menu | 筋トレの「レップ→セット→メニュー」と同型で自然。ただし Kotlin `Set` 型との衝突回避が必要 |
| stage / menu | 段階的に設定が変わる進行の語感。悪くないが決め手なし |
| segment | Strava の「区間」の連想が強く早期に除外 |
| round | 1 往復 (= 既存 cycle) の意味に読めて紛らわしく早期に除外 |
| **menu / program** | **採用**。「今日のメニュー: 坂道×5、流し×3」のように個々の練習項目を「メニュー」と呼ぶ日本語のジム語感に合致。全体は program |
| menu / routine | program と同型。routine は「毎朝の決まった流れ」の語感で、定義体の名前としては program のほうが中立と判断 |

## 決定 (2026-07-21)

- **cycle**: 高強度→回復の 1 往復 (既存のまま)。
- **menu (メニュー)**: 同一設定で cycle を N 回繰り返すまとまり。
- **program (プログラム)**: menu を順に並べたトレーニング全体の定義。実行結果である session と対になる (program = 予定・定義、session = 実行した記録)。

確定した用語は `docs/stock/requirements.md` の拡張リストに反映済み。データ構造 (menu が持つ設定の範囲、既存の単一設定 `SessionConfig` からの移行) は未設計で、次の検討対象。
