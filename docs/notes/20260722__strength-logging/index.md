# 筋トレ記録 (Workout) の設計検討

## 2026-07-22 背景

毎朝ジムで筋トレ→有酸素 HIIT の順に運動しているが、アプリは有酸素側しか扱っていない。筋トレ側 (マシン毎の負荷×回数) を phone で記録し、Firestore に分析用バックアップとして残したい、という要望。「phone = リッチ UI / 分析、watch = 1 タップ実行」という既存分担に合致する phone 専用機能として設計する。

## 用語の決定

ユーザ確定済み: **トレーニング** = 1 マシンの種目 or ストレッチ、**セット** = トレーニング内の繰り返し単位 (負荷×回数)。型名は Kotlin `Set` との衝突回避で `TrainingSet`。

複数トレーニングをまとめる単位 (ジム 1 回分) の候補比較:

| 候補 | 評価 |
|---|---|
| **Workout (ワークアウト)** | **採用**。ジム語彙として「筋トレ 1 回分」に最も自然。コードベースでは Android 定数 (`CATEGORY_WORKOUT` 等) 以外で未使用で、既存ドメイン名 (session/menu/program/cycle) と衝突しない |
| Routine (ルーティン) | 「決まった手順・計画」の含意が強く、1 回分の実行記録の名前としてはズレる。program/menu 命名検討 ([20260721__program-menu-naming](../20260721__program-menu-naming/index.md)) でも routine は定義体寄りの語として不採用だった |
| GymVisit | 意味は最も直接的だが、ジム語彙として不自然で UI 英語表記も硬い |

Firestore コレクション名 `workouts` はデータが入ると実質変更不可のため、この時点でユーザ確認して確定した。

## 主要な設計判断

- **Workout 開始時刻 = 最初の入力時** (画面 open 時ではない)。覗いただけで空レコードができるのを防ぎ、`startedAtMs` を分析上意味のある値に保つ。
- **負荷 (weightKg) は nullable**。ストレッチ・自重トレーニングに負荷概念がないため。reps は必須のまま (ストレッチの秒数対応が欲しくなったら nullable フィールド追加で拡張)。
- **名前一意性は trim + 大文字小文字無視、非表示トレーニングも含めて判定**。非表示を除くと再表示時に衝突するため。
- **ローカル永続化は DataStore (カタログ) + ファイル per レコード (workout)。Room は使わない**。カタログは `library/current` と同型の小さな LWW 文書で `PhoneLibraryRepository` パターンを流用できる。workout は追記中心・アクティブ 1 件で `PendingSessionStore` のファイルパターンが合う。Room は本 repo では表示専用キャッシュの位置づけでクエリ需要もない。
- **カタログの Firestore (`strengthCatalog/current`) は書き込み専用**。「Firestore は分析用バックアップでありアプリ機能のソースにしない」方針に従う。再インストールでカタログはローカル消失するが許容 (形は `library/current` と同じなので、将来 opt-in 復元を足す余地はある)。
- **進行中 workout も入力毎に同一 doc へ upsert**。「セット入力時点で永続化」を Firestore まで貫く。放置レコードは `endedAtMs: null` のまま残るが、分析ではむしろ「終了し忘れ」として有用。
- **自動終了はバックグラウンドタイマーなしの遅延評価**。トリガーは (1) 画面 open 時 (2) 有酸素ライブスナップショット受信時 (3) 完了 SessionRecord 受信時 (live 取り逃しの安全網) の 3 点。終了は状態管理のみでデータは保存済みのため、リアルタイム性が不要。自動終了時の `endedAtMs` は発見時刻ではなく `lastInputAtMs` (実際の運動終了に近い値)。

## 追加合意 (2026-07-22 プランレビュー)

1. トレーニング並び順は**登録順で固定**。ジムの順路どおりに登録すればそのまま順路リストになる。最近実施順は毎回並びが変わり入力が遅くなるため不採用。
2. **ジム名・トレーニング名の rename を含める** (一意性チェック共通化)。削除は hide で代替し実装しない。記録側は名前をスナップショットで持つため rename しても過去記録は不変。
3. Finish ボタン脇に**未入力トレーニング数を表示** (Finish は止めない)。「やらなかったら明示的に入力する」の運用を UI が補助する。
4. 過去 workout の閲覧画面はスコープ外 (データは残るので後から作れる)。
5. TDEE / カロリー連携はスコープ外。
6. UI 表記は "Workout" (画面タイトル・overflow メニュー)。コードも `Screen.Workout` / `WorkoutScreen` / `WorkoutActions` に揃える。カタログ型はジム/トレーニング台帳なので `StrengthCatalog`。
7. 同名トレーニングはジムごとに別エンティティ (前回値も独立)。マシンが違えば負荷も違うため。

確定仕様は `docs/stock/requirements.md`「筋トレ記録」節と `docs/stock/sync.md` の Firestore スキーマに反映する。
