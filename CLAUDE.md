# adaptive-pulse

心拍数をトリガーに高強度/回復フェーズを自動で切り替える、Wear OS (Pixel Watch) 向けアダプティブ・インターバルトレーニングアプリ。個人利用 (サイドロード) 前提の個人プロジェクト。

## 協働モデル (常に適用)

- 確認質問は出さず、最も妥当な 1 案で進める。完了報告で「悩んだ点・採用した判断」を明示し、レビューで戻せる状態にする。
- 直前の指示の論理的帰結は確認なしで実行する。
- 次の作業ステップへの着手可否を尋ねない。ロードマップ (`docs/stock/requirements.md` の「開発の進め方」と残課題) の次ステップに自動で着手する。
- 例外として確認してよいのは (1) 不可逆操作 (force push / 削除 / 外部公開) (2) 強い個人嗜好を伴う選択 のみ。

## 常に守ること

- 本 repo は **public**。コミットした瞬間に全世界へ公開される前提で扱う (`secret.md` 参照)。
- コメントは Why のみ。How は書かない。
- ユーザからの補足・指摘・決定は会話に残さず `.claude/rules/*.md` に反映する (該当が無ければ新設)。一般原則を含む指摘は、その場のインスタンスを直すだけでなくルールに昇格させる。
- ドキュメントは stock / flow に分類して配置する。
- 変更はこまめに commit & push する (push 時に hook が build/test を通し、CI が再検証する)。push 後は CI の結果を確認し、失敗していたら同ターンで修正する。

## 基本思想

- 提供価値は「自分が継続できる開発体験 (DX) と、ジムで毎朝使える運用品質」。機能追加は「DX か運用品質に資するか」で評価する。
- AI エージェント前提で設計する。判断軸: AI が読みやすい構造 (明示的な型・命名) / FB を得やすい仕組み (短いビルド・テスト、機械可読なエラー、hook 検証) / 介入しやすい粒度 (小さい関数、副作用の局所化、境界の明示) / 「なぜ・何が起きたか」を文脈として残す。
- ドメインロジック (ステートマシン) と Android フレームワークを分離し、エミュレータ・実機なしで JVM 単体テストだけで検証できる状態を保つ。
- flame (https://github.com/wakuwaku3/flame) の思想を踏襲しつつ個人開発向けに軽量化する。
- 確定した要件・技術選定は stock ドキュメント (`docs/stock/`) を参照する。

詳細な開発ルールは `.claude/rules/` にあり、Claude Code が自動ロードする (CLAUDE.md からの参照は不要)。
